# Практическая работа №5

**Цель работы:** создать функциональную корзину для товаров и создание заказов из них. Также необходимо добавить авторизацию, регистрацию, выход из профиля, разграничение прав доступа.

В рамках практической работы была реализована функциональная корзина для веб-приложения булочной «Хлебный Дом» на фреймворке Django. Основной задачей было сделать так, чтобы пользователь мог добавлять товары из каталога в корзину, изменять количество позиций, удалять товары, а затем создавать заказ на основе содержимого корзины. Дополнительно в проект были добавлены регистрация, авторизация, выход из аккаунта и разграничение прав доступа для обычных пользователей и сотрудников.

Первым этапом была реализована корзина с хранением данных через сессии пользователя. Для этого в файле `bakery/views.py` был добавлен ключ `CART_SESSION_KEY = 'cart'`, а также вспомогательные функции для получения корзины, загрузки товаров из базы данных и обработки количества товара. Корзина хранится в `request.session` в формате, где ключом является идентификатор товара, а значением - количество выбранных единиц товара. Такой подход позволяет сохранять содержимое корзины между запросами пользователя без создания отдельной таблицы в базе данных.

Рисунок №47 - Реализация хранения корзины в сессии

Для добавления товаров в корзину был создан обработчик `AddToCartView`. Он получает товар по идентификатору, проверяет его наличие, считывает количество из формы и записывает выбранную позицию в сессию. Если товар уже находится в корзине, его количество увеличивается. После изменения корзины сессия помечается как измененная через `request.session.modified = True`, чтобы Django сохранил обновленные данные.

Рисунок №48 - Добавление товара в корзину

Для управления содержимым корзины были реализованы представления `UpdateCartItemView` и `RemoveFromCartView`. Первое представление позволяет изменить количество товара в корзине: если пользователь указывает количество больше нуля, значение обновляется, а если количество равно нулю или меньше, товар удаляется. Второе представление полностью удаляет выбранную позицию из корзины. Благодаря этому корзина стала полноценной и удобной для пользователя.

Страница корзины оформлена в шаблоне `bakery/templates/bakery/cart.html`. На ней выводится таблица с товарами, количеством, ценой, суммой по каждой позиции и итоговой стоимостью заказа. Для каждой позиции предусмотрены формы обновления количества и удаления товара. Если корзина пуста, пользователь видит сообщение о пустой корзине и ссылку для перехода в каталог товаров.

Рисунок №49 - Страница корзины

Следующим этапом была реализована возможность создания заказа из корзины. В файле `bakery/views.py` создано представление `CreateOrderView`, которое доступно только авторизованным пользователям. При отправке формы оформления заказа представление получает товары из сессии, проверяет, что корзина не пустая, рассчитывает итоговую стоимость и создает запись в таблице `Order`. Заказ создается внутри транзакции, что повышает надежность операции: заказ и его позиции создаются как единое действие.

Рисунок №50 - Создание заказа из корзины

После создания заказа автоматически формируются связанные записи `OrderItem`. Для каждой позиции корзины сохраняется товар, количество и цена на момент оформления заказа. Это важно, потому что цена товара в каталоге может измениться позже, а в заказе должна сохраниться стоимость, актуальная именно на момент покупки. После успешного оформления корзина очищается, а пользователь перенаправляется на страницу созданного заказа.

Рисунок №51 - Позиции созданного заказа

Для привязки заказа к пользователю в модели `Order` используется поле `user`, связанное со стандартной моделью пользователя Django. При создании заказа выполняется привязка к текущему авторизованному пользователю через `request.user`. Благодаря этому каждый заказ имеет владельца, а в дальнейшем можно разграничивать доступ к заказам между пользователями.

Рисунок №52 - Поле пользователя в модели заказа

Отдельным этапом была добавлена регистрация пользователей. В файле `bakery/forms.py` создана форма `RegistrationForm` на основе стандартной формы `UserCreationForm`. Она позволяет пользователю указать логин, email, имя, фамилию и пароль. В файле `bakery/views.py` реализовано представление `RegisterView`, которое создает нового пользователя и сразу выполняет вход в аккаунт после успешной регистрации.

Рисунок №53 - Форма регистрации пользователя

Также была добавлена авторизация и выход из аккаунта. Для входа используется стандартное представление Django `LoginView` с шаблоном `bakery/login.html`, а для выхода - `LogoutView`. В файле `bakery/urls.py` добавлены маршруты `/login/`, `/register/` и `/logout/`. В настройках проекта `bakery_project/settings.py` прописаны параметры `LOGIN_URL`, `LOGIN_REDIRECT_URL` и `LOGOUT_REDIRECT_URL`, которые задают страницы входа и перенаправления после авторизации или выхода.

Рисунок №54 - Форма авторизации

Рисунок №55 - Выход из аккаунта

Важной частью работы стало разграничение прав доступа. Для административных действий был создан миксин `StaffRequiredMixin`, который разрешает доступ только авторизованным пользователям со статусом `is_staff`. Он применяется к представлениям создания, редактирования и удаления товаров, категорий, производителей, акций, покупателей, заказов и позиций заказов. Таким образом обычный пользователь не может попасть на страницы административного управления данными.

Рисунок №56 - Проверка прав staff-пользователя

Для заказов была реализована дополнительная проверка прав. Пользователь с правами staff может видеть все заказы, а обычный авторизованный пользователь видит только свои заказы. Для этого в `OrderListView` переопределен метод `get_queryset`: если пользователь является сотрудником, возвращаются все заказы, иначе выполняется фильтрация по полю `user`. Для страницы детального просмотра заказа используется проверка, разрешающая доступ владельцу заказа или staff-пользователю.

Рисунок №57 - Ограничение просмотра заказов

Разграничение прав доступа также выполнено на уровне шаблонов. В базовом шаблоне `bakery/templates/bakery/base.html` настроен условный вывод элементов навигации. Если пользователь авторизован, в хедере отображается только кнопка выхода из аккаунта. Если пользователь не авторизован, отображаются кнопки авторизации и регистрации. Административные ссылки на покупателей, заказы и позиции заказов показываются только staff-пользователям. Для обычного авторизованного пользователя дополнительно выводится ссылка «Мои заказы».

В шаблонах списков и детальных страниц также добавлены условия для отображения кнопок действий. Кнопки добавления, изменения и удаления сущностей выводятся только пользователям с правами staff. Кнопка добавления отзыва доступна только авторизованным пользователям. За счет этого интерфейс не показывает пользователю действия, которые ему недоступны по правам.

Рисунок №61 - Условный вывод кнопок в шаблонах

После реализации функционала была выполнена проверка работы проекта. Были запущены команды `python manage.py check` и `python manage.py test`. Системная проверка Django не выявила ошибок, а автоматические тесты подтвердили корректную работу добавления товара в корзину, создания заказа, привязки заказа к пользователю, очистки корзины после оформления, переключения кнопок авторизации в хедере и ограничения просмотра заказов для обычного пользователя.

Рисунок №62 - Успешное выполнение тестов

**Вывод:** в ходе выполнения практической работы была реализована полноценная логика корзины и оформления заказов для интернет-магазина булочной на Django. Корзина хранит данные в сессии пользователя, товары можно добавлять, удалять и изменять их количество. Из корзины создается заказ, который автоматически привязывается к авторизованному пользователю и содержит связанные позиции заказа с количеством и ценой товара на момент покупки. Дополнительно были реализованы регистрация, авторизация и выход из аккаунта. Для защиты данных настроено разграничение прав доступа: staff-пользователь управляет справочниками и видит все заказы, а обычный пользователь видит только собственные заказы. В интерфейсе элементы навигации и кнопки действий отображаются в зависимости от статуса авторизации и прав пользователя. В результате проект стал более функциональным, безопасным и готовым к дальнейшему развитию.
